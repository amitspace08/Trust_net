Task 4: Week 22nd  June -26th  June 

1. What This Week Is About 

Week 3 gave you the trust circle — users can now find each other, connect, and see each 
other on the map. This week you build the reason the entire app exists. 

The SOS system. 

By Friday, pressing the SOS button on one phone must result in every Layer 1 contact 
receiving a push notification with a live map showing exactly where that person is right now. 
A contact must be able to tap the notification, open a screen, and see the distressed user's pin 
moving in real time. 

This is the core of TrustNet. Everything else — the heatmap, the social graph, the Guardian 
Angels — sits on top of what you build this week. If the SOS does not work by Friday, 
nothing else matters. 

Task 1 — The SOS Trigger and Cancel Window 

Goal: One button press starts the emergency. Five seconds to cancel. After that, the system 
takes over automatically. 

S1 — Frontend 

1. Build the SOS button — large, red, centre of the home screen. It must be impossible 
to miss but hard to press accidentally. Use a long-press of 1.5 seconds to activate, not 
a single tap. 

2. Build the 5-second cancel screen — full screen red, countdown from 5 to 0 with a 
large circular progress animation, and a clearly visible "Cancel — I am safe" button. 
If the user does nothing for 5 seconds, SOS activates automatically. 

3. Build the SOS Active screen — shown to the distressed user after activation. It must 
show: current status (which layer is being alerted), a live map of their own location, 
and a large "I am safe — end SOS" button at the bottom. 

4. Build the SOS Receiver screen — shown to a contact who receives an SOS 
notification and opens the app. It must show: the distressed user's name, their live 
location on a full-screen map, a moving pin, and a large "I am responding — I am on 
my way" button. 

5. Make sure the cancel button on the 5-second screen is large enough to tap under 
stress. This is not a normal UI task — the person using this screen may be frightened. 
Design accordingly. 

 

S2 — Backend 



1. Design and create the sos_sessions Firestore collection with the following fields: 
sessionId, triggeredBy (uid), status (active / cancelled / resolved), layerActive (1, 2, or 
3), startTime, endTime, layer1Alerted (array of uids), layer1Acknowledged (uid or 
null). 

2. Write the triggerSOS(uid) function — creates a new SOS session document, sets 
status to active, sets layerActive to 1, records startTime, and returns the sessionId. 

3. Write the cancelSOS(sessionId) function — sets status to cancelled and records 
endTime. Must work within the 5-second window. 

4. Write the acknowledgeSOS(sessionId, responderUID) function — records who is 
responding, sets layer1Acknowledged, and updates status to resolved. 

5. Write the endSOS(sessionId) function — called when the distressed user taps "I am 
safe". Sets status to resolved and records endTime. 

6. Write a Firestore security rule ensuring only the session owner can cancel or end their 
own SOS session. 

 

S3 — Maps 

1. During an active SOS session, start sending the distressed user's GPS location to a 
live_locations Firestore document every 5 seconds — faster than the normal 30-
second interval used in Week 3. 

2. On the SOS Receiver screen, subscribe to the distressed user's live_locations 
document using onSnapshot so the map pin updates in real time without any manual 
refresh. 

3. Draw a pulsing red circle animation around the distressed user's pin on the receiver's 
map — the circle expands and fades repeatedly to communicate urgency. 

4. Show the distance from the responder to the distressed user in the bottom bar of the 
receiver screen, updating every 5 seconds. 

5. When the SOS ends, stop the location updates immediately and clear the 
live_locations document. 

 

S4 — Features 

1. Fetch the triggering user's Layer 1 contact list using S2's getLayer1Contacts function. 

2. Send a push notification to every Layer 1 contact using Firebase Cloud Messaging. 
The notification must include: the distressed user's name, a short message ("Priya has 
triggered an SOS — she needs help now"), and a deep link that opens the SOS 
Receiver screen directly. 



3. Start a 90-second countdown timer from the moment Layer 1 is alerted. If no one 
acknowledges within 90 seconds, log "Layer 1 timeout — ready for Layer 2 
escalation" to the console. Layer 2 itself will be built in Week 5 — for now, just prove 
the timeout logic works. 

4. Handle the case where a contact receives the notification but has the app closed. The 
notification must still arrive and open the correct screen when tapped. 

5. Test the full end-to-end flow with all 4 team phones. One phone triggers SOS, the 
other three receive the notification, one of them acknowledges, confirm the session 
status updates to resolved in Firestore. 

 

Task 2 — Live Location Tracking During SOS 

Goal: The distressed user's location must update every 5 seconds on the receiver's map for the 
entire duration of the SOS session. This must work even when the distressed user's phone 
screen is off. 

S1 — Frontend 

1. On the SOS Active screen, show a small live status bar at the top: "Location sharing 
— live" with a pulsing green dot. 

2. Show who has acknowledged in real time — when a contact taps "I am responding", 
their name should appear on the distressed user's screen within 3 seconds. 

3. Add a timeline at the bottom of the SOS Active screen showing: SOS triggered at 
[time] → Layer 1 alerted → [contact name] responded at [time]. 

4. When the SOS ends, transition smoothly to a "You are safe" confirmation screen with 
the option to rate the response. 

 

S2 — Backend 

1. Add a responders subcollection inside each sos_sessions document to record: uid, 
name, respondedAt, and currentLocation of each person who acknowledged. 

2. Write the updateResponderLocation(sessionId, uid, lat, lng) function — called every 
10 seconds by each responding contact to share their location back to the distressed 
user. 

3. Ensure the sos_sessions document has a real-time listener trigger — when 
layer1Acknowledged is updated, the distressed user's screen must reflect this within 2 
seconds. 

 

S3 — Maps 



1. On the SOS Active screen (the distressed user's view), show the responder's location 
as a blue pin moving toward them once a contact acknowledges. 

2. Draw a live route line from the responder's current location to the distressed user, 
updating as the responder moves. 

3. Show estimated time of arrival (ETA) from the responder's pin to the distressed user's 
pin using straight-line distance divided by average walking speed (1.4 metres per 
second) as a rough estimate. 

S4 — Features 

1. Implement background location tracking so GPS updates continue every 5 seconds 
even when the distressed user's screen is locked. Use Expo's background location task 
for this. 

2. Send a follow-up push notification to unacknowledged Layer 1 contacts at the 45-
second mark — "Priya still needs help — please respond now." 

3. When the 90-second Layer 1 timeout fires, update the sos_sessions document: set 
layerActive to 2 and add a layer1Timeout timestamp. This sets up next week's Layer 2 
implementation. 

4. Test background location — trigger SOS, lock the sending phone, walk around, 
confirm the pin moves on the receiving phone. 

 

Task 3 — The "I Am Safe" Flow 

Goal: The distressed user must be able to cancel or end an SOS cleanly, and all active 
responders must be notified immediately. 

S1 — Frontend 

1. The "I am safe — end SOS" button must require a double-tap to confirm — one 
accidental tap should not end an active SOS. 

2. After ending, show a brief post-SOS screen: "SOS ended. You are safe. [Responder 
name] was on the way." with a thank-you message for the responder. 

3. If a contact is mid-journey when the SOS ends, they must see a notification on their 
SOS Receiver screen: "Priya has marked herself safe. Thank you for responding." 

S2 — Backend 

1. When endSOS is called, send a notification to all active responders that the 
emergency has ended. 

2. Archive the completed SOS session — move it from sos_sessions to a sos_history 
collection with all fields intact. This data will be used later for the Guardian Angel 
response history feature. 



3. Clear the distressed user's live_locations document immediately when SOS ends. 

S3 — Maps 

1. When SOS ends, smoothly fade out the pulsing red circle and the route line on all 
receiver screens. 

2. Show a brief "All clear" banner on the receiver's map before returning them to the 
normal map view. 

S4 — Features 

1. Cancel all pending FCM notifications for this SOS session when it ends — contacts 
who have not yet opened the notification should not see a stale alert. 

2. Test edge case: distressed user's phone loses internet during active SOS. What 
happens to the live location feed? Document what breaks and propose a fix — you do 
not need to implement it this week. 

Demonstration scenario: One team member plays the distressed user. The other three play 
contacts. The distressed user presses and holds the SOS button → 5-second cancel screen 
appears → countdown completes → all three other phones receive a push notification → one 
opens the app and sees the live moving pin → taps "I am responding" → the distressed user's 
screen shows the responder's name → the distressed user taps "I am safe" → all phones 
receive the all-clear notification.
