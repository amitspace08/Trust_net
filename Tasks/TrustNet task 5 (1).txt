Task 5: Week 29th  June – 11th July 

1. What This Week Is About 

Task 4 gave you a working SOS system. Layer 1 contacts get alerted, they see a live pin, they 
respond, and the session closes cleanly. That is good. But it only solves the easy case — when 
someone picks up. 

This week you solve the hard case. What happens when nobody responds? 

You will build Layer 2 — the friends-of-friends escalation. When 90 seconds pass and no Layer 
1 contact responds, TrustNet must automatically traverse the social graph, identify second-
degree contacts ranked by proximity, and alert them with a notification that establishes implicit 
trust through the mutual connection. 

This is the feature that makes TrustNet worth patenting. No other safety app does this. The 
students building it this week are building something genuinely new. 

By Friday, the following must work on real phones: SOS triggers, Layer 1 gets alerted, 90 
seconds pass with no response, the system automatically identifies and alerts friends-of-friends, 
one of them responds, session resolves. End to end, no manual steps. 

Understand the Social Graph 

Read this carefully before writing any code. 

• A social graph is a map of relationships. Every user is a node. Every trust connection is 
an edge. When Priya has Mum, Riya, and Arjun as Layer 1 contacts, there are three 
edges from Priya's node outward. 

• Graph distance means how many hops it takes to reach someone. Priya's Layer 1 
contacts are at distance 1 — one hop. Their contacts who are not already Priya's contacts 
are at distance 2 — two hops. These are the Layer 2 candidates. 

Task 1 — Social Graph Traversal Engine 

Goal: Given a distressed user's UID, the system must identify, deduplicate, and rank all second-
degree contacts within 30 seconds of the Layer 1 timeout firing. 

S1 — Frontend 

1. Build the Layer 2 escalation status screen — this is shown to the distressed user after 
the 90-second Layer 1 timeout. It must show: "No response from your contacts. Finding 
nearby people who know them..." with a subtle animated searching indicator. Do not 
make this screen alarming — the user is already scared. Keep it calm and reassuring. 

2. Build the Layer 2 alert notification screen — shown to a Layer 2 contact who opens the 
app after receiving an alert. It must show: "A friend of [Contact Name] needs help 
nearby", the distressed user's approximate location on a map (not exact — fuzzy within 
200m), and a large "I can help — I am on my way" button. 



3. Add a "Sent as friend of [Name]" label clearly visible on the Layer 2 receiver screen. 
This is the trust signal — the Layer 2 contact needs to know they are not being contacted 
by a stranger completely out of nowhere. They are being contacted because someone 
they know knows the person in trouble. 

4. Update the escalation timeline on the SOS Active screen to include: "Layer 1 alerted 
→ No response in 90s → Layer 2 alerted → [Name] responded". 

5. The Layer 2 receiver screen must look visually distinct from the Layer 1 receiver screen 
built in Task 4. Use a different header colour or badge to make it clear this is a different 
level of alert. 

S2 — Backend 

1. Write the getLayer2Candidates(uid) function. This is the core of this week. It must:  

o Fetch the user's Layer 1 contact UIDs from their user document 

o For each Layer 1 contact, fetch their Layer 1 array 

o Combine all results into a single array 

o Remove the original user's own UID 

o Remove any UIDs already in the original user's Layer 1 

o Remove duplicates 

o Return the final deduplicated array of Layer 2 candidate UIDs 

2. Write the rankLayer2Candidates(candidateUIDs, distressedUserLocation) function. It 
must:  

o Fetch each candidate's current location and availability from Firestore 

o Filter out candidates whose available field is false 

o Sort remaining candidates by straight-line distance from the distressed user 
(closest first) 

o Return the top 10 ranked candidates with their UID, name, phone, and distance 

3. Write the triggerLayer2(sessionId) function. It must:  

o Call getLayer2Candidates to get the candidate list 

o Call rankLayer2Candidates to sort them 

o Update the sos_sessions document: set layerActive to 2, add layer2Alerted 
array, add layer2TriggerTime timestamp 

o Return the ranked list for S4 to use for notification dispatch 



4. Write the acknowledgeLayer2(sessionId, responderUID) function — same logic as 
Layer 1 acknowledgement but updates layer2Acknowledged field. 

5. Add a 120-second timeout trigger for Layer 2 — if no Layer 2 contact acknowledges 
within 120 seconds, update layerActive to 3 and add a layer2Timeout timestamp. Layer 
3 will be built in Week 7 — for now just prove the timeout fires correctly. 

6. Update Firestore security rules to allow Layer 2 candidates to read an SOS session 
document when they have been added to the layer2Alerted array. 

S3 — Maps 

1. On the Layer 2 receiver screen, show the distressed user's location as a fuzzy pin — not 
a precise point. Implement location fuzzing by adding a random offset of between 100 
and 200 metres to the actual coordinates before displaying them. The Layer 2 contact 
sees approximately where the person is, not exactly. Exact location is only revealed 
after they confirm they are responding. 

2. Once a Layer 2 contact taps "I can help", replace the fuzzy pin with the precise live 
location pin and begin live tracking exactly as in Task 4. 

3. On the distressed user's SOS Active screen, show Layer 2 candidates as teal-coloured 
pins on the map — distinct from the red user pin and the blue Layer 1 responder pins 
used in Task 4. Label each pin "Layer 2 — [first name]". 

4. Show a coverage radius circle on the map — a visible circle of approximately 500m 
around the distressed user showing the geographic zone being searched for Layer 2 
candidates. This gives the distressed user a sense that the system is actively looking, 
not stuck. 

5. When a Layer 2 contact confirms they are responding, animate their pin moving from 
teal to blue and begin drawing the route line toward the distressed user, exactly as done 
in Task 4 for Layer 1. 

S4 — Features 

1. Wire the existing 90-second Layer 1 timeout (built in Task 4) to call S2's triggerLayer2 
function automatically when it fires. This is the connection between Task 4 and Task 5 
— the timeout that was logging to console last week must now actually do something. 

2. Send Layer 2 push notifications to the ranked candidate list returned by triggerLayer2. 
The notification body must include: "A friend of [Layer 1 contact name] needs help 
near you. Can you assist?" — the mutual connection name must be personalised for 
each recipient. If Dev is a friend of Mum, Dev's notification says "A friend of Mum". 
If Anita is a friend of Riya, Anita's notification says "A friend of Riya." 

3. The personalised notification requires knowing which Layer 1 contact links the 
distressed user to each Layer 2 candidate. Modify getLayer2Candidates to also return 
this linking contact's name for each candidate so the notification can use it. 



4. Start the 120-second Layer 2 countdown timer from the moment Layer 2 notifications 
are sent. Log "Layer 2 timeout — ready for Layer 3 escalation" to console when it 
expires. 

5. Test the full escalation chain: trigger SOS → 90 seconds pass → Layer 2 fires → Layer 
2 candidate receives personalised notification → they respond → session resolves. 
Document this test with screenshots of each step. 

6. Test with a minimum of 5 user accounts so the graph traversal has real data to work 
with. Create test accounts for: Priya (distressed user), Mum (L1), Riya (L1), Dev 
(Mum's contact = L2), Anita (Riya's contact = L2). 

Task 2 — Trust Signal and Notification Design 

Goal: A Layer 2 contact is a semi-stranger. The notification they receive must give them 
enough context to trust the situation and act on it — without revealing unnecessary personal 
information about the distressed user. 

S1 — Frontend 

1. Design the Layer 2 notification carefully. It must answer three questions the recipient 
will immediately have: who needs help, why are you being contacted, and where do 
you need to go. The current template "A friend of [Name] needs help nearby" answers 
the first two. Add a distance indicator — "approximately 300m from you" — to answer 
the third. 

2. Build a pre-response confirmation screen for Layer 2 contacts — before they can tap "I 
can help", show them: the mutual connection's name, approximate distance, and a brief 
one-line explanation of TrustNet ("You are registered as a community helper. Someone 
nearby needs assistance."). This prevents accidental confirmations. 

3. Add a "I cannot help right now" button alongside "I can help" — Layer 2 contacts must 
be able to decline. When declined, the system should immediately try the next ranked 
candidate. 

4. Build a "Thank you" screen shown to a Layer 2 contact after the SOS is resolved — 
"You helped someone feel safer today. Thank you." This is important for community 
retention — people who feel appreciated come back. 

S2 — Backend 

1. Handle the decline flow — write a declineLayer2(sessionId, responderUID) function 
that removes the declining UID from the active candidates list and triggers notification 
to the next ranked candidate. 

2. Ensure the ranking function never sends to the same person twice in one session — if 
they decline, they are removed from the candidate pool for that session permanently. 

3. Write a getMutualConnection(userAUID, userBUID) function — given the distressed 
user and a Layer 2 candidate, returns the name of the Layer 1 contact that connects 



them. This is used by S4 for personalised notifications and by S1 for the pre-response 
screen. 

S3 — Maps 

1. When a Layer 2 candidate declines, smoothly remove their teal pin from the distressed 
user's map and add the next candidate's pin. 

2. Add a small counter on the distressed user's SOS Active screen showing "Checking 
nearby people: 3 contacted, 1 declined, 1 responding." This gives the distressed user 
visibility into what the system is doing. 

S4 — Features 

1. Implement sequential notification logic for declined candidates — when declineLayer2 
is called, immediately send the notification to the next person in the ranked list without 
waiting. The candidate pool should drain from top to bottom based on rank. 

2. Test the decline flow: Layer 2 candidate A declines → immediately candidate B 
receives notification → B accepts → session resolves. Confirm this on real phones. 

3. Test edge case: all Layer 2 candidates decline or are unavailable. Confirm that the 120-
second timeout still fires correctly and layerActive is set to 3 in Firestore. 

Task 3 — Privacy in Layer 2 

Goal: Layer 2 contacts are not people Priya chose. She did not give them permission to know 
her location. The system must handle her location data carefully at this layer. 

S1 — Frontend 

1. Add a one-line privacy notice on the SOS Active screen when Layer 2 activates: 
"Showing your approximate location only. Exact location shared when someone 
confirms they are helping." 

2. Show the distressed user a list of who is being alerted in Layer 2 — name and mutual 
connection only, not phone number. "Dev (friend of Mum) — 280m away." This gives 
the distressed user awareness and some sense of control. 

S2 — Backend 

1. Ensure the layer2Alerted array in Firestore only stores UIDs, not names or phone 
numbers, to limit data exposure in the session document. 

2. When a Layer 2 candidate is alerted, write their UID to a separate active_layer2_alerts 
collection with a TTL (time to live) of 10 minutes — if the session resolves or they do 
not respond within 10 minutes, this document is automatically deleted. 

3. Never write the distressed user's precise coordinates to any document visible to Layer 
2 candidates. Only the fuzzy coordinates computed by S3 should appear in their 
receiver screen data. 



S4 — Features 

1. When the SOS session ends (either resolved or cancelled), immediately send a deletion 
trigger to clear all active_layer2_alerts documents related to that session. Layer 2 
candidates who did not respond should have no remaining data about the session in any 
document they can access. 

2. Test and confirm after SOS ends, a Layer 2 candidate who did not respond cannot query 
any Firestore document that contains the distressed user's location or name. Document 
this test result. 

NOTE: 

• The mutual connection name is non-negotiable. The personalised notification is the 
entire trust mechanism. "Someone near you needs help" is ignored. "A friend of Mum 
needs help near you" is acted on. Every test must verify the correct name appears. 

• Fuzzy location must actually be fuzzy. Do not show the precise pin to Layer 2 
contacts before they accept. This is a privacy commitment that is also part of the patent 
claims. Test it — look at what coordinates are sent to the Layer 2 receiver screen and 
confirm they are offset. 

• Test with 5 accounts minimum. The graph traversal only works correctly when there 
is actual graph data. Four team phones are not enough — create one extra Firebase test 
account manually and add it to the graph.
