Task 6: Week 12th – 19th July 

1. What This Week Is About 

Weeks 4 and 5 built Layer 1 and Layer 2 of the SOS escalation. This week you complete the 
system with Layer 3 — the Guardian Angel network. You are also building Community Safe 
Spaces and a basic safety rating feature that shows users how safe their current location is right 
now. 

By end of this week the core TrustNet product is complete. All three SOS layers run 
automatically in sequence. Safe Spaces appear on the map. Users can rate their current area 
and see a safety score badge. Everything that was described in the original TrustNet concept 
exists as working code. 

This is the building week. Week 7 is the finishing and presentation week. Do not leave anything 
unbuilt hoping to finish it next week — Week 7 has no time for new features. 

 

Task 1 — Guardian Angel Registration and Profile 

Goal: Any TrustNet user can register as a Guardian Angel — a verified community helper who 
agrees to receive Layer 3 SOS alerts when they are nearby and available. 

 

S1 — Frontend 

1. Build the Guardian Angel registration screen — accessible from the Profile screen via 
a "Become a Guardian Angel" button. Show a brief explanation of what the role 
involves, a confirmation checkbox, and a Register button. Keep it simple — this is a 
one-time setup screen. 

2. Build the Guardian Angel Dashboard — shown to registered Guardian Angels on their 
Profile screen. The most prominent element must be the availability toggle — large, 
clearly ON or OFF, impossible to miss. Below it show total responses, star rating, and 
a short response history list. 

3. Build the Guardian Angel Profile screen — shown to a distressed user when a Guardian 
Angel is matched to their SOS. Show name, photo, verification badge, star rating, 
number of past responses, and current distance. 

4. Build the Layer 3 Alert screen — shown to a Guardian Angel when they receive a Layer 
3 notification and open the app. Show "Someone nearby needs help urgently", 
approximate distance, a fuzzy location map, and two buttons — "I can help — on my 
way" and "I cannot help right now." 

5. Update the SOS Active screen to show the current active layer at all times using distinct 
colours — purple for Layer 1, teal for Layer 2, gold for Layer 3. Add a simple escalation 
timeline that fills in as events happen. 

 

S2 — Backend 



1. Add Guardian Angel fields to the user document in Firestore: isGuardianAngel 
(boolean), guardianVerified (boolean), guardianRating (number), 
guardianResponseCount (integer), guardianAvailable (boolean), guardianRegisteredAt 
(timestamp). 

2. Write the registerAsGuardianAngel(uid) function — sets isGuardianAngel to true, 
guardianVerified to true (automatic for internship demo), guardianAvailable to true. 

3. Write the setGuardianAvailability(uid, available) function — updates 
guardianAvailable instantly when the toggle is changed. 

4. Write the findGuardianAngels(lat, lng, radiusMetres) function — queries Firestore for 
all users where isGuardianAngel is true AND guardianAvailable is true AND location 
is within the given radius. Return results sorted by distance. If no results are found at 
500m, automatically expand to 750m, then 1000m. 

5. Write the triggerLayer3(sessionId) function — calls findGuardianAngels, updates the 
sos_sessions document with layerActive set to 3, layer3Alerted array, and 
layer3TriggerTime timestamp. 

6. Write acknowledgeLayer3(sessionId, guardianUID) and declineLayer3(sessionId, 
guardianUID) functions. 

7. Write updateGuardianRating(guardianUID, rating) — updates guardianRating as a 
running average and increments guardianResponseCount. Called after a session ends. 

 

S3 — Maps 

1. Show available Guardian Angels as gold shield pins on the main map — visually 
distinct from contact pins (purple) and Safe Space pins (green, built in Task 2). Only 
show Guardian Angels within 2km of the user. 

2. On the Layer 3 Alert screen, show the distressed user's fuzzy location as a pulsing gold 
circle — same fuzzy offset logic used in Week 5 for Layer 2. 

3. Once a Guardian Angel confirms they are responding, switch their pin from gold to 
blue and draw a live route line from their location to the distressed user — same pattern 
as Week 4 Layer 1 response. 

4. When Layer 3 activates on the distressed user's SOS Active screen, draw a visible 500m 
radius gold dashed circle around their location and show available Guardian Angel pins 
inside it. 

5. When a Guardian Angel declines, fade their pin to grey and animate the next candidate's 
pin appearing gold. This gives the distressed user a real-time visual of the system 
working through candidates. 

 

S4 — Features 

1. Wire the Layer 2 120-second timeout (built in Week 5, currently logging to console) to 
call S2's triggerLayer3 function automatically. This single connection completes the 
full three-layer chain. 

2. Send Layer 3 push notifications to Guardian Angels with: "Emergency nearby — 
someone needs help [X]m from you. Can you respond?" Include a 60-second response 
window — shorter than Layer 2 because Guardian Angels signed up specifically for 
this role. 



3. Implement sequential notification for declined Guardian Angels — when one declines, 
immediately notify the next in the ranked list without waiting for a new timeout. 

4. If all Guardian Angels within the expanded radius decline or are unavailable, update 
the session document with layer3Exhausted set to true and show the distressed user the 
nearest Safe Space as the primary call to action. 

5. Build a post-SOS rating prompt — after the session ends, ask the distressed user to rate 
their Guardian Angel from 1 to 5 stars. Pass the result to S2's updateGuardianRating 
function. 

Task 2 — Community Safe Spaces 

Goal: Registered businesses and public locations appear on the map as Safe Spaces. During 
any active SOS session — regardless of which layer is active — the nearest Safe Space is 
always shown to the distressed user as a physical fallback destination. 

 

S1 — Frontend 

1. Build the Safe Space registration screen for business owners — collect business name, 
type (shop, pharmacy, petrol station, police station, public building), and confirm GPS 
location. Keep it under 5 fields total. 

2. Show Safe Space pins on the main map as green building icons — distinct from all 
other pin types. 

3. Build the Safe Space detail popup — shown when a user taps a Safe Space pin. Show 
name, type, distance, and the badge "TrustNet Safe Space — you can walk in and ask 
for help here." 

4. During any active SOS session, show a persistent banner at the bottom of the SOS 
Active screen: "Nearest safe space: [Name] — [X]m away" with a tap-to-navigate 
button that opens walking directions in Google Maps. 

5. Build a "Find Safe Spaces" list screen — shows all Safe Spaces within 1km sorted by 
distance. Accessible from the home screen via a small button. 

 

S2 — Backend 

1. Create a safe_spaces Firestore collection with fields: spaceId, name, type, geopoint, 
address, verified (boolean), registeredAt (timestamp). 

2. Write registerSafeSpace(name, type, lat, lng, address) function. 
3. Write getNearestSafeSpace(lat, lng) function — returns the closest verified Safe Space. 

Called at the start of every SOS session and whenever the distressed user's location 
updates significantly. 

4. Write getSafeSpacesWithinRadius(lat, lng, radiusKm) function — used by the Find 
Safe Spaces screen. 

5. Seed 15 dummy Safe Spaces around Jaipur using real location names — include police 
stations, 24-hour pharmacies, petrol stations, and malls. This gives the map instant 
credibility in the demo. 

 



S3 — Maps 

1. Render Safe Space pins as green building icons on the main map. Ensure they are 
visually distinct from user pins, contact pins, and Guardian Angel pins at a glance. 

2. During an active SOS session, highlight the nearest Safe Space with a pulsing green 
border animation — more prominent than its normal static pin. 

3. Draw a dashed green walking route from the distressed user's location to the nearest 
Safe Space during SOS. This gives the user a clear visual path to physical safety if no 
responder has confirmed. 

 

S4 — Features 

1. During an active SOS session, continuously update the nearest Safe Space banner as 
the distressed user moves — if they walk closer to a different Safe Space, update 
automatically. 

2. When layer3Exhausted is set to true (all Guardian Angels declined), make the Safe 
Space banner the most prominent element on the screen — increase its size and add a 
message: "Walk to the nearest safe space now. Help is registered there." 

3. Test: trigger SOS, let it escalate fully through all three layers with no responses, confirm 
the Safe Space becomes the primary call to action on screen. 

Task 3 — Basic Safety Rating and Score Badge 

Goal: Users can rate the safety of their current location on a scale of 1 to 10. A simple coloured 
badge on the map shows the average safety score for wherever the user currently is. No full 
heatmap grid. No Cloud Functions. No temporal decay. Just submit a score and see an average. 

 

S1 — Frontend 

1. Add a floating safety score badge on the map screen — a small coloured circle in the 
corner showing the current area's average safety score. Green for 7 and above, amber 
for 4 to 6, red for 3 and below, grey if no data exists yet. 

2. Build the Rate This Area screen — accessible by tapping the badge or via a "Rate this 
area" button. Show a 1-to-10 score selector, optional tags (Poor lighting, Isolated, Well-
lit, Crowded, Police presence), and a Submit button. The entire screen must be 
completable in under 20 seconds. 

3. After submitting, show a brief confirmation: "Your rating helps keep your community 
safe." Then return to the map automatically. 

4. When the user taps the badge, show a small popup: current average score, number of 
ratings, and the top two tags reported for this area. 

S2 — Backend 

1. Create a safety_ratings Firestore collection with fields: ratingId, uid, geopoint, score 
(1-10), tags (array), timestamp, gridCell (string — computed from lat/lng to group 
nearby ratings together). 



2. Write submitRating(uid, lat, lng, score, tags) function — validates input, computes 
gridCell, writes to safety_ratings. 

3. Write getAreaScore(lat, lng) function — queries all safety_ratings within 150m of the 
given coordinates, calculates a simple unweighted average, and returns the score and 
rating count. If fewer than 3 ratings exist, return null. 

4. Prevent rating abuse — a user cannot submit more than one rating for the same gridCell 
within 2 hours. Check this before writing. 

S3 — Maps 

1. When the map screen loads, call S2's getAreaScore with the user's current location and 
update the floating badge colour and number. 

2. Refresh the badge every time the user moves more than 150m — fetch the new area 
score for their updated location. 

3. If getAreaScore returns null (fewer than 3 ratings), show the badge in grey with a 
question mark and the tooltip "No data yet for this area — be the first to rate it." 

S4 — Features 

1. Write a seed script that generates 80 realistic dummy safety ratings around Jaipur — 
enough to make the badge show real scores in at least 5 to 6 different areas during 
testing rather than always showing grey. 

2. Prevent a single user's ratings from dominating an area score — cap each user's 
contribution to any gridCell at 3 ratings per 24 hours regardless of how many they 
submit. 

3. Test: submit a rating, immediately call getAreaScore for the same location, confirm the 
score updates. Submit 3 more ratings from different accounts, confirm the average 
reflects all of them. 

Week 6 Deliverables 

Demonstration scenario 1 — Full three-layer chain: Trigger SOS. Layer 1 times out at 90 
seconds. Layer 2 fires automatically, times out at 120 seconds. Layer 3 fires automatically. 
Guardian Angel receives notification. Accepts. Session resolves. Total time under 215 seconds. 

Demonstration scenario 2 — Safe Spaces: Open the map. Show 15 Safe Space pins visible 
around the area. Trigger SOS. Nearest Safe Space banner appears on the SOS Active screen 
within 5 seconds. Tap the banner — walking directions open in Google Maps. 

Demonstration scenario 3 — Safety badge: Open the map screen. Safety score badge shows 
in corner. Tap it — popup shows score and tags. Submit a new rating from one phone — score 
updates on a second phone within 30 seconds of submission.
